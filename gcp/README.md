# GCP layout policy

`gcp/` is the single deployable surface for VM and bucket.

**Source of truth:**
- **`gcp/stage/`** — Local staging area, 1:1 mirror of bucket root (runners, configs, results, triggers, etc.). Edited locally then synced to GCS via `just deploy`. `gcp/stage/` is under local control — never directly mounted via FUSE.
- **`gcp/image/`** — VM code and config; **managed via infra** (Packer). Contents are baked into the VM image at build time; they are not uploaded to the bucket. Publish code = image build from this repo.
- **`gcp/mnt/`** — FUSE observation window (gitignored). Optional read-only gcsfuse mounts for browsing GCS input data locally. See *Data access tiers* below.

Allowed top-level entries:

- `gcp/image/` — VM code/config (infra-managed; baked into image)
- `gcp/stage/` — Local staging area (1:1 with bucket root, replaces former `gcp/remote/`)
- `gcp/mnt/` — FUSE mount points (gitignored, opt-in)
- `gcp/README.md`

Any extra top-level directory is policy drift.

## What belongs where

- **gcp/image:** Watcher, orchestrator, VM lib/config/scripts, and per-project managers. All of this is **managed via infra**: Packer ([infra/packer/bmt-runtime.pkr.hcl](../infra/packer/bmt-runtime.pkr.hcl)) bakes it into the VM image. Do not upload gcp/image to the bucket; publish = image build.
- **gcp/image/schemas/:** Versioned JSON schemas for **runtime-generated** artifacts. See [gcp/image/schemas/README.md](image/schemas/README.md).
- **gcp/stage:** Runner binaries, input placeholders (`.keep`), `dataset_manifest.json` files, configs, and any other bucket-mirrored content. This is the local staging view of bucket content.
  - `gcp/stage/projects/<name>/inputs/<dataset>/.keep` — placeholder; keeps the dir visible in git.
  - `gcp/stage/projects/<name>/inputs/<dataset>/dataset_manifest.json` — tracked in git; describes the dataset (file names, sizes, checksums). See *Data access tiers*.
- **gcp/mnt:** FUSE mount points, gitignored entirely. Never a sync source.
- **Runner lib dependencies:** Shared native deps live in `gcp/local/dependencies/`. Each project’s lib dir (`gcp/local/<project>/lib/`) should contain `libKardome.so` plus symlinks to shared deps. Run `just symlink-deps` to create/refresh.
- Do not put `triggers/`, `results/`, or `outputs/` under `gcp/stage` (generated at runtime).
- Do not put real WAV datasets under `gcp/stage/**/inputs` (up to 30-40 GB; upload explicitly via `just upload-data`).
- Do not mount `gcp/stage/` via FUSE — gcsfuse warns against VCS repos on FUSE mounts.
- Do not put `__pycache__` or `*.pyc` under either mirror.

## Data access tiers

Three tiers of local data access, in increasing fidelity/complexity:

| Tier | How | What you see | Online? |
| :--- | :-- | :----------- | :------ |
| **1 — Manifest only** | `git clone` (zero deps) | File names + sizes in `dataset_manifest.json` | No |
| **2 — On-demand fetch** | `just fetch-inputs <project> <dataset>` | Real files under `gcp/stage/projects/<p>/inputs/<d>/` | Yes |
| **3 — FUSE mount** | `just mount-data <project>` | Full read-only filesystem tree at `gcp/mnt/<p>-inputs/` | Yes |

### Manifest contract

`dataset_manifest.json` is generated by `just gen-manifest <project> <dataset>` and tracked in git.  It describes a dataset at a point in time:

```json
{
  "schema_version": 1,
  "project": "sk",
  "dataset": "false_rejects",
  "bucket": "my-bmt-bucket",
  "prefix": "projects/sk/inputs/false_rejects",
  "generated_at": "2026-03-15T12:00:00Z",
  "files": [
    {"name": "ambient/cafe_001.wav", "size_bytes": 4812800, "sha256": "abc123...", "updated": "..."}
  ]
}
```

Regenerate after uploading a dataset: `just gen-manifest sk false_rejects` then commit.

### FUSE mount (opt-in, dev QoL)

```bash
just mount-data sk        # mount gs://$GCS_BUCKET/projects/sk/inputs → gcp/mnt/sk-inputs/
just umount-data sk       # unmount
```

**WSL2 pitfalls (kernel 6.6):**
- `chmod`/`chown` silently fail on gcsfuse mounts. Tools must not call `os.chmod()` on mounted paths.
- The mount is torn down when the WSL session exits. Re-mount with `just mount-data` per session.
- Add `gcp/mnt/` to IDE ignore lists so the IDE does not index WAV files.
- Heavy traversal without `--stat-cache-ttl` floods the GCS List API. The `just mount-data` recipe uses `--stat-cache-ttl=300s`.

**Never mount `gcp/stage/` via FUSE** — git uses `flock`, hardlinks, and atomic renames that are unsupported on gcsfuse.

## Manual operations

- Sync staging area to GCS: `just deploy`
- Fetch a full dataset locally: `just fetch-inputs <project> <dataset>`
- Fetch a single file: `just fetch-wav <gcs-relative-path>`
- Regenerate dataset manifest: `just gen-manifest <project> <dataset>`
- Validate local layout policy: `just test` or `uv run python -m tools repo validate-layout`
- Pre-commit hook blocks commits that touch `gcp/` unless the bucket is in sync (or `SKIP_SYNC_VERIFY=1`). Run `just deploy` to sync first.
- Local diagnostics are non-authoritative and belong under `.local/diagnostics/`.
